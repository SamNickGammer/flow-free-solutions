# 06 — UX: what the app actually does

> Interactive mockup: **https://claude.ai/code/artifact/82891bbd-dac8-40bc-bdee-a411bc036a5c**
> (tap the bubble, then pick a mode)

This is the product spec Phase 4 and Phase 5 build against. It is the user's design, written down
so it doesn't get lost.

---

## The core rule: solve silently, then ask

**Tapping the bubble does not change anything on screen.** It captures, reads the grid, and solves —
and then it **shows you nothing**. The solution is held in memory. The board stays clean.

Then it asks you one question, and you choose:

```
tap bubble  →  capture  →  read + solve  →  ┌─ ASK YOU ─┐  →  ⚡ Draw it  (app draws it, flow by flow)
   (you)      (1 frame)    (<1s, silent)   │ board is  │      ✋ Manual   (faint overlay + arrows,
                                            │  still    │                  you trace it yourself)
                                            │  clean    │
                                            └───────────┘
```

**Nothing is drawn on the board until you have picked a mode.** Dumping every flow on screen the
instant you tap takes the choice away — and half the time, drawing it yourself is the entire point.

---

## Mode: ⚡ Draw it (auto)

The app draws the solution **itself**, one flow at a time, like a hand.

### Yes, Android can genuinely do this

An **`AccessibilityService`** can call **`dispatchGesture()`** with a `Path`, and Android
synthesises **real touch events**. The game cannot distinguish them from a finger. **No root
required.** (API 24+.)

### How

1. Each solved flow is an ordered list of cells.
2. Convert cells → screen pixels using the detected grid geometry
   (`screen_px = board_origin + cell_index * cell_size + cell_size/2`).
3. Build **one `Path`** through the cell centres.
4. Dispatch as a **single stroke**: press at the first dot, drag through every cell, lift at the
   last.
5. One stroke per flow, **dispatched in sequence** — visibly, one by one, so the user can watch it
   happen.

It looks like a hand drawing it because that is precisely what it is.

### Tuning knobs that will actually matter

- **Stroke speed.** Too fast and the game drops cells mid-drag. Needs a real, user-visible knob.
  Start slow (**~40–60 ms per cell**) and tighten only once proven on the device.
- **Interpolation.** Sample points **per cell**, not just at path corners, or a long straight run
  can skip cells.

*(ponytail: one duration-per-cell setting, exposed in settings. The physical device needs tuning a
minimal model can't predict — leave the knob.)*

### The honest risks

| Risk | Consequence |
|---|---|
| **Calibration off by a few px** | A swipe starts on the wrong cell — drags a flow you didn't mean to and **wrecks a board you'd half-finished.** Destructive, not just wrong. |
| **Game sets `FLAG_SECURE`** | `MediaProjection` returns a **black frame**. Flow Free doesn't today, but it's one release away and there is **no workaround**. |

**Because a mis-aimed swipe is destructive, "Draw it" must only fire on a board the detector read
*confidently*.** See [open question 1](#open-questions).

## Mode: ✋ Manual (mild overlay)

The brief was **mild**, and that is the whole design. This is **not** the answer painted over the
game. It's a **hint layer you can see through**, telling you which way to move.

| Element | Why |
|---|---|
| **Faint path** — ~35% opacity, thinner than the real pipe | You must still see the game's own board and your own progress *underneath* it. |
| **Direction arrows** — chevrons along each path | A line alone doesn't say *which end to start from*. Arrows do. |
| **Pulsing start dot** | Tells your finger exactly where to land first. |
| **Touch-transparent** (`FLAG_NOT_TOUCHABLE`) | You are tracing *through* the layer. |

You then trace it on the real game with your own finger, and the level solves itself as you go.

---

## ⚠️ The constraint that makes or breaks the whole product

> **The overlay window MUST be touch-transparent — `FLAG_NOT_TOUCHABLE`.**

If it is touchable, it **swallows your finger and you cannot trace the answer it is showing you**.

The app would look flawless in a screenshot and be **completely unusable in the hand**. This is the
kind of bug that survives every test that isn't a human holding a phone.

**Only the bubble takes touches. Everything else is `FLAG_NOT_TOUCHABLE`.**

## Permissions

| Permission | For what | Required? |
|---|---|---|
| **Draw over other apps** (`SYSTEM_ALERT_WINDOW`) | The bubble + the hint overlay | **Yes** |
| **Record the screen** (`MediaProjection`) | One frame per solve | **Yes** |
| **Accessibility** (`dispatchGesture`) | Auto-draw only | **No — optional** |

**Accessibility is optional and must stay that way.** Manual mode works without it. Never make
someone grant the scariest permission to get the *safer* feature.

---

## Failure state: say so, don't guess

Reading the grid is the riskiest part of this project — themes, notches, DPI, ad banners, and the
[wall-vs-hole distinction](levels/03-walls.md) all conspire against the detector.

When it misreads, it must **say so** and let the user fix it in two seconds: show the detected grid
with **draggable corners**, report what it found (`5×5, 4 colours, 1 wall`), and remember the
correction.

A confidently wrong answer is worse than an honest "I couldn't read this" — and **with auto-draw
armed, it is actively destructive.**

> The free **parity pre-check** ([04-solver-design.md](04-solver-design.md)) catches many detector
> misreads in O(nodes) before any search runs. Use it as the confidence gate for auto-draw.

---

## Decided

- **Solve silently, then ask.** Keeps the choice — and the game — in the user's hands.
- **Bubble, not a notification.** A notification means leaving the game to tap it. The entire point
  is not leaving the game.
- **Accessibility optional.** Manual mode never requires it.
- **No account, no network. Fully on-device.** Screen-capture data must never leave the phone, and
  nothing here needs a server.

## Open questions

1. **Auto-draw: confirm first, or undo after?** A mis-aimed swipe drags the wrong flow and wrecks a
   half-finished board. Preview the first stroke before committing — or fire and offer a reset?
2. **Manual: all flows at once, or one at a time?** Four sets of arrows may be visually busy.
   Revealing one flow per tap keeps it a puzzle and still unsticks you. Cheap either way — the
   solver already returns paths per colour.
3. **Should the bubble know it's Flow Free?** Watching the foreground app lets it auto-hide
   everywhere else — costs another permission.
4. **Name.** "Flow Free Assist" is a placeholder.
